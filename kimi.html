EXECUTIVE SUMMARY

DETAILED FINDINGS

상세 이슈 및 개선점

6.1 P0 — 즉시 수정 필요(기능 결함)

P0 lib/processor.ts 테이블명 불일치

runPipelineAndSave 함수가 process_pipelines, process_runs 테이블을 참조하지만, 실제 마이그레이션 011_processing.sql은 processing_pipelines, processing_runs를 생성한다. 따라서 이 함수를 호출하면 DB 오류가 발생한다.

// lib/processor.ts (line 390, 398)
await supabase.from('process_pipelines').select('*')...
await supabase.from('process_runs').insert({ ... })

개선: lib/processor.ts의 모든 테이블 참조를 processing_pipelines/processing_runs로 수정. 참고로 app/api/process/* 경로는 올바른 테이블명을 사용하고 있어, 라이브러리와 API 간 불일치 상태다.

P0 __tests__/lib/evaluation.test.ts 동기화 실패

computeAiReadyChecklist은 8개 항목을 { name, pass, detail } 형태로 반환하지만, 테스트는 5개 항목과 { item, passed, detail } 프로퍼티를 기대한다. 결과적으로 npm test가 3개 테스트에서 실패한다.

// __tests__/lib/evaluation.test.ts line 47-49
it('체크리스트가 5개 항목', () => {
  expect(checklist).toHaveLength(5) // 실제는 8개
})

// line 72-78
expect(c).toHaveProperty('item') // 실제는 name
expect(c).toHaveProperty('passed') // 실제는 pass

개선: 테스트를 현재 구현에 맞게 8개 항목 / name·pass·detail 프로퍼티로 수정하거나, 반대로 구현을 테스트 기대에 맞춘다. UI에서 사용하는 프로퍼티명과도 일치해야 한다.

6.2 P1 — 보안 및 운영 위험

P1 하드코딩된 평문 암호화 키

lib/collector.ts에서 환경변수 COLLECTION_SECRET이 없을 경우 평문 기본값을 사용한다. 이 값이 그대로 커밋되어 있으므로, 누구든 암호화된 auth_value를 복호화할 수 있다.

// lib/collector.ts line 11
const SECRET = (process.env.COLLECTION_SECRET ?? 'eum_default_secret_key_32bytes!!').padEnd(32, '!').slice(0, 32)

개선: 기본값 제거 후 환경변수 누락 시 애플리케이션 기동/호출 실패 처리. 기존 암호화 값은 즉시 재암호화(키 로테이션).

P1 테스트 계정 비밀번호 하드코딩

run_seed_and_user.mjs와 scripts/create_test_users.py에 center@eum.test / eum2026!, changwon@eum.test / eum2026!이 평문으로 하드코딩되어 있다.

// run_seed_and_user.mjs line 68-69
email: 'center@eum.test',
password: 'eum2026!'

개선: 비밀번호를 환경변수로 분리하거나, 랜덤 생성 후 안전한 경로에만 기록. .gitignore에 test_accounts.txt는 이미 등록되어 있으나 스크립트 자체가 노출되어 있다.

P1 예측 가능한 난수 ID 생성

lib/processor.ts의 runPipelineAndSave가 Math.random()으로 run_id와 dataset_id를 생성한다. 충돌 가능성이 있고 보안 식별자로 부적합하다.

// lib/processor.ts line 396, 420
const runId = 'prun_' + Math.random().toString(36).slice(2, 10)
const datasetId = 'proc_' + Math.random().toString(36).slice(2, 10)

개선: crypto.randomUUID() 또는 lib/utils.ts::randomHex를 사용. app/api/process/[id]/run/route.ts는 이미 crypto.getRandomValues를 사용하므로 이 패턴으로 통일.

P1 Middleware가 /api/* 전체를 공개로 처리

middleware.ts의 PUBLIC_PREFIXES에 '/api/'가 포함되어 있어, 미들웨어 레벨에서 모든 API 경로에 인증 검사를 수행하지 않는다. 각 API Route가 개별적으로 인증해야 하며, 누락 시 보안 허점이 발생한다.

// middleware.ts line 5
const PUBLIC_PREFIXES = ['/login', '/api/']

개선: 공개 API 목록을 /api/public/*, /api/v1/datasets/*, /api/dcat 등으로 명시적으로 한정하거나, 각 API Route의 인증 여부를 전수 점검하고 문서화한다.

6.3 P2 — 유지보수 및 코드 품질

P2 next.config.js Server Actions origin 불일치

allowedOrigins: ['localhost:3000']으로 설정되어 있지만, npm run dev는 3001 포트를 사용한다. 로컬 개발 시 Server Actions가 원본 검증으로 거부될 수 있다.

개선: localhost:3001로 변경하거나, 환경변수/개발 모드에 따라 동적으로 설정.

P2 중복된 Python 실행 래퍼

app/api/analyze/route.ts와 app/api/analyze/upload/route.ts에 동일한 runAnalyzePy 헬퍼가 중복 구현되어 있다.

개선: lib/analyzeRunner.ts 등으로 추출하여 중복 제거 및 단위 테스트 용이하게 개선.

P2 lib/quality.ts의 as any 연산자 호출

countWhere 함수에서 필터 연산자를 (q as any)[op](col, val)로 동적 호출한다. 런타임에 잘못된 연산자가 들어오면 오류가 발생하며, 타입 안정성도 없다.

// lib/quality.ts line 41-43
const q = sb.from(table).select('*', { count: 'exact', head: true })
const { count } = await (q as any)[op](col, val)

개선: 허용 연산자 화이트리스트와 타입 오버로드를 추가하여 안전하게 호출.

P2 서비스롤 클라이언트에 불필요한 쿠키 핸들러

lib/supabase/server.ts의 createServiceClient가 서비스롤 키를 사용하면서도 사용자 쿠키 핸들러를 그대로 전달한다. 서비스롤은 세션이 필요 없으므로 혼란과 잠재적 버그를 야기할 수 있다.

개선: 서비스롤 클라이언트는 { auth: { persistSession: false } } 등 쿠키 없는 설정을 사용하거나, 용도를 명확히 문서화한다. lib/openapi.ts::createPublicClient가 이미 그러한 패턴을 보여준다.

P2 ESLint 미설정

npm run lint 실행 시 ESLint 설정이 없어 "How would you like to configure ESLint?" 프롬프트가 표시된다. 코드 품질 자동 검사가 동작하지 않는다.

보안·인증

P1 일부 API Route가 세션 없이 민감 동작 수행

middleware.ts가 /api/* 전체를 공개로 처리하는 가욱, 아래 Route는 사용자 인증 없이 호출될 수 있다.

Route현재 상태위험도
GET /api/quality인증 없이 runAll(supabase) 실행 → 품질진단 배치중간(자원 낭비·데이터 노출)
POST /api/ontology인증 없이 buildOntology(supabase) 실행 → 그래프 재구축중간(부하·데이터 갱신)
POST /api/chat인증 없이 Qwen API 프록시 → 외부 API 비용/남용높음(비용·콘텐츠 정책)
GET /api/nlquery인증 없이 자연어 질의 → 데이터 접근낮음~중간(공개 데이터 기준)
GET /api/overview인증 없이 대시보드 요약 조회낮음(공개 요약 데이터)

권고: 각 Route의 의도를 재확인하고, 인증이 필요한 Route에는 supabase.auth.getUser() 체크를 추가한다. 공개 API는 /api/public/* 또는 /api/v1/*로 명확히 분리한다.

  2. Phase 10: 보안·성능·관측성 기반 — DEV_BYPASS 제거, 서비스롤 남용 해소, 분산 rate limit, SQL 샌드박스, 구조화 로깅/트레이싱을 구축한다. 운영 배포 전 필수 기반.
  3. Phase 9: 내구성 있는 데이터 파이프라인 — 실제 커넥터 런타임, 스트리밍 ETL, 작업 큐/오케스트레이션, 스키마 레지스트리, 데이터 계보를 완성한다.
  4. Phase 8: 그래프 네이티브 온톨로지 — temporal/provenance, 그래프 DB(Cypher/GQL), 임베딩 기반 KG, 관계 추론을 도입한다.
  5. Phase 11: UX·모바일·아키텍처 완성 — 서버 우선 아키텍처, 디자인 시스템, PWA, 모바일 퍼스트 재설계, 테스트 격상을 마무리한다.

Phase 7: 엔터프라이즈 AI/데이터 인텔리전스

  • Production RAG 스택: 벡터 + FTS/BM25 + structured-filter fusion(RRF/learned ranker), cross-encoder reranker, 실제 데이터 행/컬럼 프로파일/온톨로지 이웃 요약을 컨텍스트에 포함. 검색 평가용 golden question/feedback 테이블 도입.
  • Qwen 전용 LLM 게이트웨이: Qwen 호환 API/local fallback, JSON Schema structured generation, 사용자/테넌트별 quota, prompt versioning, trace, 비용/지연 메트릭. 외부 공급자(OpenAI/Anthropic)는 도입하지 않는다.
  • NL-to-SQL 에이전트 완성: 스키마 인트로스펙션 + 샘플 행 + 온톨로지 정의 + few-shot 예제 제공, AST 기반 SQL 검증, tenant RLS 준수 read-only sandbox, 실행 실패 시 self-correction loop.
  • 비동기 임베딩 파이프라인: catalog/ontology 변경에 반응하는 queue/worker, batched embedding, HNSW 인덱스, 모델-차원 검증.
  • AI 안전성: PII redaction(presidio), prompt-injection 탐지, 출력 moderation, 환각/근거성 평가, 대화 메모리 서버 이전·테넌트 격리.

Phase 8: 그래프 네이티브 온톨로지 & 지식 그래프

  • Temporal/provenance-aware KG: valid_from/valid_to, source_dataset, etl_run_id, version를 노드/엣지에 추가하고 히스토리 스냅샷 유지.
  • Graph-native engine: pg_graph/Apache AGE 또는 외부 그래프 DB 도입, Cypher/GQL 쿼리, 확장 가능한 traversal.
  • 증분/스냅샷 재빌드: 현재 삭제 후 삽입 방식 대신 변경 감지 기반 증분 업데이트, 롤백 가능한 스냅샷.
  • Graph embedding & KG completion: 노드 임베딩, 링크 예측, LLM+규칙 기반 관계 추론, 메타데이터로부터의 자동 엣지 제안.
  • 고급 분석: 중심성/커뮤니티/이상탐지 메트릭 사전 계산, 시나리오 시뮬레이션, 영향 분석(impact analysis).

Phase 9: 내구성 있는 데이터 파이프라인 & 운영

  • 실제 커넥터 런타임: Postgres 증분 커서/CDC, SFTP 파일 폴링+스트림, API/Webhook 연동. connector_config를 collector.ts가 사용하도록 통합.
  • 스트리밍·청크 ETL: Generator/AsyncIterable, bulk INSERT/COPY, 체크포인트/워터마크, 장애 시 resume.
  • 작업 큐 + 오케스트레이션: pgmq/BullMQ/Redis 기반 큐, 멱등성 키, 동시 실행 제한, exponential backoff, dead-letter queue, 취소 API.
  • 스키마 레지스트리 + 데이터 계약: 자동 프로파일링, 버저닝, quarantine, quality_contract를 수집 단계에서 강제.
  • 데이터 거버넌스: raw/staging/derived 계층, retention/archiving, lineage DAG 시각화, 영향 분석, audit 집계.
  • Enterprise 공개 API: API 키별 quota, IP allowlist, dataset scope, versioning, 분산 rate limit.

Phase 10: 보안·성능·관측성 기반

  • Zero-trust 인가: DEV_BYPASS 완전 제거, PUBLIC_PREFIXES 축소, Policy-as-Code 중앙화, 서비스롤 남용 최소화, mTLS/서명 JWT 기반 서비스 간 인증.
  • 실행 샌드박스화: run_select_sql 대신 AST 기반 안전 쿼리 빌더, catalog.table_name 화이트리스트 RPC, Python subprocess Docker/WASM 격리, LLM PII 마스킹.
  • 분산 rate limit & WAF: Redis/Upstash 기반 sliding window, tiered limit, Vercel Firewall/Cloudflare 연계, body size/file upload 검증.
  • Observability: Pino/Winston 구조화 로그, OpenTelemetry trace_id, Sentry, Deep Health Check, Prometheus metrics, Grafana/Vercel Monitoring, SLI/SLO.
  • 보안 헤더 강화: nonce 기반 CSP(unsafe-inline/eval 제거), HSTS preload, Permissions-Policy, COOP, report-uri.
  • 비밀 관리 & 테스트: Vault/Doppler, Argon2id API key 해시, rotation, SAST/DAST(Semgrep/ZAP), 보안 단위 테스트, 의존성 취약점 검사.

Phase 11: UX·모바일·아키텍처 완성

  • 서버 우선 아키텍처: React Server Component + Supabase SSR, TanStack Query 캐싱/낙관적 업데이트, Zod API 스키마, ok/error 응답 봉투 통일.
  • 엔터프라이즈 디자인 시스템: CSS 변수/토큰, 인라인 스타일 제거, Radix/shadcn 기반 컴포넌트, Storybook + Chromatic, axe CI를 error로 격상.
  • 모바일 네이티브: 모바일 퍼스트 재설계, 하단 시트/낭게이션, 터치 타겟 44×44, PWA + offline cache, pull-to-refresh.
  • 실시간/알림 레이어: 중앙 이벤트 버스, Server Actions 뮤테이션, 글로벌 알림 센터.
  • 성능 & 테스트: 번들 분석/성능 예산, 동적 임포트, Web Vitals/Sentry, E2E 모바일/태블릿, 컴포넌트 테스트, 스크린샷 회귀.
