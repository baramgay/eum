# 이음(EUM) 플랫폼 — 공급자 워크플로우 설계

> 1단계 MVP(합성데이터·읽기전용 5탭) 다음 사이클. 마스터플랜의 "공공데이터 개방 지원" 영역 중
> "기관이 데이터를 등록→표준화→품질점검→개방 결정"하는 공급자 워크플로우를 설계한다.
> "활용·확산 촉진"은 이 워크플로우가 갖춰진 다음 사이클에서 별도로 다룬다.
>
> **디자인 방향**: 이번 사이클은 신규 기능을 *기존 디자인 시스템(`web/style.css`) 그대로*
> 구현해 일관성과 검증 용이성을 우선한다. "플랫폼 전체 애플 스타일 재디자인"은 기능이
> 안정화된 뒤 별도 스펙·계획으로 분리해 진행한다(아래 "범위 밖" 참조) — 기능과 디자인을
> 한 번에 바꾸면 회귀를 가려내기 어렵기 때문이다.

## 배경 및 목적

기존 MVP는 미리 적재된 합성 데이터를 보여주는 읽기 전용 플랫폼이었다. 플랫폼의 핵심 목적인
"공공데이터 개방 확대"를 실현하려면, 각 기관(시군 등)의 담당자가 직접 데이터를 등록하고
품질을 점검받은 뒤 스스로 개방 여부를 결정할 수 있는 기능이 필요하다. 센터(이음 운영자)는
이 과정의 승인권자가 아니라 — **전체 현황을 모니터링하고 컨설팅 의견을 제공하는 지원자**다.
이 점이 설계 전반의 권한 구조를 결정한다.

## 역할 모델

워크플로우에는 두 역할만 둔다 (간단한 모드 전환, 로그인 없음 — 기존 MVP와 동일한 데모 수준 검증 방식).

| 역할 | 누구 | 권한 |
|---|---|---|
| 기관 담당자 | 데이터를 등록하는 시군·기관 실무자 | 등록·진단 확인·**스스로 승인/반려 결정** |
| 센터(컨설팅) | 이음 플랫폼 운영자(경남빅데이터센터) | 전체 현황 모니터링, 컨설팅 코멘트 작성 — 승인 권한 없음 |

## 워크플로우 흐름

```
[기관 담당자 모드]
  ① 파일 업로드(CSV) → 스키마 추론 → DuckDB 테이블 생성 → 미리보기 표시
  ② 메타정보 입력 (제목·설명·주제·키워드·라이선스 — DCAT 필드)
  ③ "등록" → 자동으로 품질·표준화 진단 실행, 결과 즉시 표시 (status = submitted)
  ④ 담당자가 진단 결과를 검토하고 본인이 직접 결정:
       → "승인(개방)"  : catalog 등록 → 개방포털 노출 (status = approved)
       → "반려(보류)"  : 내부 보관, 메모 가능 (status = rejected)
     ※ 두 상태 모두 양방향 전환 가능 — 진단 보완 후 재검토·재승인 가능

[센터(컨설팅) 모드]
  - 전체 기관의 등록 현황·진단 결과·개방 비율을 모니터링 (대시보드)
  - 특정 제출 건에 컨설팅 코멘트(개선 의견)를 남겨 담당자에게 전달
  - 승인/반려에는 관여하지 않음
```

설계 포인트:
- `draft`(임시저장) 단계는 두지 않는다. 업로드 즉시 미리보기까지 보여준 뒤 "등록" 시점에
  `submitted`로 저장한다 — 불필요한 중간 상태를 늘리지 않는다.
- 진단은 등록 시점에 자동 실행되어, 담당자가 승인/반려를 결정할 때 이미 결과가 준비돼 있다.
- 반려(재검토) 시 기존 레코드의 상태만 바꾸고 이력을 보존한다 — 별도의 재제출 레코드를
  만들지 않아 데이터 모델이 단순해진다.

## 데이터 모델

```sql
submissions
  submission_id   TEXT PK
  tenant_id       TEXT          -- 등록 기관(시군)
  title, description, theme, keywords, license, format   -- DCAT 메타
  table_name      TEXT          -- 적재된 DuckDB 테이블명
  rows            INTEGER
  status          TEXT          -- submitted / approved / rejected
  quality_summary TEXT          -- 자동 진단 결과 요약(오류율·통과 규칙 수 등)
  decision_note   TEXT          -- 담당자가 승인/반려 시 남기는 메모(선택)
  submitted_at    TEXT
  decided_at      TEXT

consultant_comments
  comment_id      TEXT PK
  submission_id   TEXT          -- FK → submissions
  comment         TEXT
  created_at      TEXT
```

`consultant_comments`를 별도 테이블로 둔 이유: 같은 제출 건에 여러 차례 의견을 남길 수
있고, 제출 이력(승인/반려 변경)과 독립적으로 누적·조회되어야 하기 때문이다.

## 범용 품질 규칙 (임의 데이터셋 자동 진단)

기존 `quality.py`의 데이터셋별 고정 규칙(`RULES`)은 미리 알려진 3개 데모 테이블에
맞춰 작성되어 임의로 업로드되는 데이터에는 적용할 수 없다. 따라서 컬럼 구조를 스캔해
자동으로 규칙을 생성하는 범용 세트를 별도로 둔다.

```python
RULES_GENERIC = [
    "결측치 비율"   : 컬럼별 NULL 비율이 임계치 초과 여부,
    "중복행 비율"   : 전체 행 대비 중복 행 비율,
    "타입 일관성"   : 숫자형으로 추정된 컬럼에 비숫자 값 혼재 여부,
    "음수 이상치"   : 건수·금액·인구 등 0 이상이어야 할 컬럼의 음수 존재 여부,
]
```

`run_quality()`가 기존 `RULES`와 동일한 인터페이스(`{rule, sql, threshold}` →
`{checked, errors, error_rate, passed}`)로 결과를 만들도록 구현해 기존 진단 파이프라인을
그대로 재사용한다. `RULES_GENERIC`을 리스트 상수로 분리해두면, 추후 특정 주제(인구·재정 등)에
특화된 규칙을 옵션으로 추가하기만 하면 적용되는 구조가 된다 — 지금 그 규칙들을 미리
구현하지는 않는다.

## 계획서 작성 지원 — 평가대응 엔진 확장

평가편람(`★2026년 공공데이터 제공 및 데이터기반행정 평가편람`)에는 기관이 직접 작성·제출해야
하는 계획서가 다수 등장한다(`27~'29 공공데이터 개방계획, 데이터기반행정 활성화 기본계획,
품질관리 평가대상 DB 선정계획, 품질진단 개선계획 등). 이 중 **이번 워크플로우에서 쌓이는
실제 데이터(제출 현황·진단 결과)로 곧바로 초안을 만들 수 있는 두 가지**를 규칙기반으로
자동 생성해 지원한다 — 마스터플랜의 "⊗ 평가대응 엔진"을 실제 기능으로 구체화하는 것이다.

| 계획서 유형 | 초안 생성 근거 | 내용 |
|---|---|---|
| 공공데이터 개방계획 초안 | `submissions`의 기관별 등록·개방·보류 현황 | 등록/개방/보류 건수, 보류 사유, 향후 개방 예정 목록을 표 형태로 자동 구성 |
| 품질개선계획 초안 | `quality_summary`(자동 진단 결과) | 오류가 발생한 규칙·오류율을 분석해 개선과제·조치방향을 규칙기반으로 제안 |

- 신규 모듈 `app/planning.py`: `draft_open_plan(tenant_id)`, `draft_quality_plan(tenant_id)`
  — 실제 등록·진단 데이터를 평가편람 양식에 맞춰 텍스트로 채운다.
- API: `GET /api/plan/draft?tenant_id=&type=open|quality`
- UI: "데이터 등록·관리" 탭의 **기관 담당자 모드**에 "계획서 초안 생성" 섹션을 추가한다
  — 유형 선택 → 초안 표시(담당자가 검토 후 다듬어 제출) → 텍스트 복사/다운로드.

기존 `nlquery.py`와 같이 **외부 LLM에 의존하지 않는 규칙기반 템플릿 채우기** 방식으로
구현해, 공공망(폐쇄망) 환경에서도 동일하게 동작하도록 한다.

## API 확장 (`main.py`)

| 메서드/경로 | 설명 |
|---|---|
| `POST /api/submission/upload` | CSV 업로드 → 스키마 추론 → DuckDB 테이블 생성 → 미리보기 반환 |
| `POST /api/submission` | 메타정보 포함 제출 생성 → 자동 품질진단 실행 → `submitted` |
| `GET /api/submission?tenant_id=` | 특정 기관의 제출 목록 (담당자 모드) |
| `GET /api/submission/all` | 전체 제출 목록 (센터 모니터링용) |
| `POST /api/submission/{id}/decision` | 담당자의 승인/반려 결정 기록 (`approved`/`rejected` + `decision_note`) |
| `POST /api/submission/{id}/comment` | 센터의 컨설팅 코멘트 작성 |
| `GET /api/submission/{id}` | 제출 상세(메타+미리보기+진단결과+코멘트 이력) |
| `GET /api/plan/draft?tenant_id=&type=` | 계획서 초안 생성 (`type=open`\\|`quality`) |

## 화면 구성 — 새 탭 "데이터 등록·관리"

상단에 `[기관 담당자 모드] / [센터(컨설팅) 모드]` 토글을 둔다(기존 SPA의
탭/뱃지 패턴을 따름).

**기관 담당자 모드**
- "새 데이터 등록": 업로드 폼 → 스키마·미리보기 표시 → 메타정보 입력 → 등록
- "내 제출 현황": 상태별(검토대기/개방/보류) 목록 → 항목 클릭 시 진단결과·승인/반려
  버튼·센터 코멘트 확인

**센터(컨설팅) 모드**
- "전체 현황": 기관별 등록 수·진단 통과율·개방 비율 요약(기존 대시보드의 카드·바 패턴 재사용)
- "전체 제출 목록"(읽기전용): 항목 클릭 시 상세 + 컨설팅 코멘트 작성

## 모듈 변경 요약

- 신규: `app/submission.py` — 파일 파싱·스키마 추론·적재·상태 관리
- 신규: `app/planning.py` — 계획서 초안 생성(개방계획·품질개선계획)
- 확장: `app/quality.py` — `RULES_GENERIC`, `generic_rules(table_name)` 추가
- 확장: `app/database.py` — `submissions`, `consultant_comments` 테이블 스키마 추가
- 확장: `app/main.py` — 위 API 8종 라우팅 추가 (제출 7종 + 계획서 초안 1종)
- 확장: `web/index.html`, `web/app.js`, `web/style.css` — 새 탭 "데이터 등록·관리"
  (워크플로우 화면 + 계획서 초안 생성 섹션)

## 범위 밖 (다음 사이클)

- 활용·확산 촉진(활용 대시보드, 활용 사례 추천, 공공이용 통계)
- 외부 공공데이터포털 API 연동 등록
- 실제 로그인/계정 기반 권한 분리
- 도메인 특화(주제별) 품질 규칙 확장
- 계획서 초안의 추가 유형(데이터기반행정 활성화 기본계획, DB 선정계획 등) 확장
- **플랫폼 전체 애플 스타일 재디자인** — 이번 기능이 안정화된 뒤, 기존 5탭 포함 전체
  디자인 시스템(`web/style.css` 등)을 별도 스펙으로 재설계. 완성된 기능 위에 새 디자인을
  입히는 순서로 진행해 회귀 위험을 줄인다.
